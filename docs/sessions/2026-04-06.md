## Bugs & Fixes (chronological)

### 1. Endpoint test fails (CORS)
**Problem**: Frontend `fetch` direct to `http://192.168.86.29:52415` blocked by browser CORS policy
**Solution**: Route through backend `/api/chat/active-model` instead of direct fetch

### 2. Nodes show 0/3 online
**Problem**: Backend used `sshExec` (SSH + curl) to query exo API. Docker container had no SSH keys to Mac Studios
**Solution**: Replace ALL `sshExec` + curl calls with direct HTTP `fetch` for exo API endpoints (status, metrics, load, unload, purge). SSH only for filesystem operations.

### 3. IP typo in settings
**Problem**: User entered `193.168.86.x` instead of `192.168.86.x` in Settings UI
**Solution**: Fixed directly in `data/settings.json` via Python on rpi-dev

### 4. Chat models shows 101 models
**Problem**: exo `/v1/models` returns full catalog including non-downloaded models
**Solution**: Parse `/state` → `downloads` entries, filter `DownloadCompleted` only. Returns 6 installed models instead of 101.

### 5. `getConfig is not a function`
**Problem**: `scanner.js` uses `getConfig` which doesn't exist in ExoScopy settings
**Solution**: Replace `scanAllNodes('exo')` with direct HTTP `/v1/models` and `/state` calls

### 6. TTFT 51.6s (Qwen3.5 thinking bug)
**Problem**: Backend only sent `enable_thinking: true` when thinking was on. When off, nothing was sent. Qwen3.5 activates thinking by default → TTFT 30-50s
**Solution**: Always send `enable_thinking: false` explicitly. TTFT dropped to 3.3s.
**Known pitfall from Model Manager** (documented in ARCHITECTURE.md #9)

### 7. Chat metrics not showing
**Problem**: Stats stored in global `lastStats` state, lost on re-render when conversation list refreshed
**Solution**: Store stats in `statsMapRef` (useRef) keyed by `convId-msgIdx`, survives re-renders

### 8. Clipboard copy fails
**Problem**: `navigator.clipboard.writeText()` requires HTTPS. rpi-dev serves HTTP
**Solution**: Fallback function using `document.execCommand('copy')` with hidden textarea

### 9. Markdown not rendering (### visible as text)
**Problem**: Local `marked.min.js` v9.1.6 uses ES modules, doesn't define `window.marked` global
**Solution**: Switch to CDN `marked@12.0.2` which provides proper UMD/global export

### 10. `model is not defined` crash
**Problem**: Typo in sync log: `model.id` instead of `modelId`. Server crashed, all subsequent requests failed
**Solution**: Fixed to `modelId`

### 11. rsync "source and destination cannot both be remote"
**Problem**: rsync can't do remote-to-remote (`source:path target:path`)
**Solution**: SSH into source node, run rsync from there to target: `ssh source 'rsync path/ target:path/'`

### 12. SSH inter-node "Host key verification failed"
**Problem**: Mac Studios didn't have SSH keys between each other (only container → nodes)
**Solution**: Enhanced "Setup SSH Keys" to install inter-node keys: generates key on each node, copies to all others via sshpass

### 13. Deleted models still show in matrix
**Problem**: exo `/state` keeps `DownloadCompleted` entries even after `rm -rf` of model files
**Solution**: Server maintains `deletedModels` Set, filters them out from matrix and chat model list

### 14. Model size shows "0 GB" for small models
**Problem**: Backend used `Math.round(bytes / 1GB)` — 327MB rounded to 0
**Solution**: Send raw bytes, frontend `formatBytes()` shows MB for <1GB, GB for larger

## Infrastructure notes

### rpi-dev (192.168.86.18)
- Raspberry Pi 5, 8GB RAM, 469GB NVMe
- Docker 29.2.1
- Open WebUI on port 3000, ExoScopy on port 3456
- SSH key from max-64 installed (`admin` user)

### Docker container
- `node:20-alpine` + `openssh-client openssh-keygen sshpass rsync`
- `~/.ssh` mounted read-write (for key generation)
- `./data` mounted for persistence (settings, conversations, downloads)
- SSH keys installed to all 3 Mac Studios + inter-node keys

### EXO cluster
- ultra-512 (192.168.86.29) — 512GB RAM, master
- ultra-256a (192.168.86.30) — 256GB RAM
- ultra-256b (192.168.86.31) — 256GB RAM
- ultra-256c (192.168.86.32) — offline during sessions
- All running exo with RDMA (Thunderbolt 5)

### Key architectural decisions
- **HTTP first for exo API** — no SSH needed for status, metrics, load/unload, model list
- **SSH for filesystem** — rsync sync, model delete, RAM stats
- **exo /state as source of truth** — DownloadCompleted = installed models, nodeNetwork = peer→IP mapping
- **deletedModels in-memory Set** — compensates for exo not updating /state after rm
